## code conventions

### naming rules

**types:**
- bunches: lowercase, no underscores (`usr`, `config`, `httpreq`)
- moods: lowercase, no underscores (`status`, `ecode`, `color`)
- og types: lowercase (`u64`, `str`, `bool`)

**variables:**
- lowercase only (`count`, `username`, `data`)
- no camelcase, no pascalcase, no caps
- underscores allowed for clarity (`user_count`, `max_retries`)

**functions:**
- lowercase only (`getuser`, `parse`, `validate`)
- no underscores preferred (`readfile` not `read_file`)
- exception: when clarity demands it (`http_get`, `db_connect`)

**mods (modules):**
- lowercase only (`net`, `http`, `crypto`)
- short names preferred (`fs` not `filesystem`)

**files:**
- lowercase only (`main.sl`, `srvr.sl`, `conf.sl`)
- shortened forms preferred (`srvr` not `server`, `conf` not `config`)
- extension: always `.sl`

**bundles (packages):**
- lowercase only
- descriptive (`httpserver`, `dbclient`, `cryptoutil`)

---

### capitalization

**zero capitals allowed:**
- no caps in code: `bunch usr` not `Bunch User`
- no caps in identifiers: `username` not `userName`
- no acronym caps: even for types like `httpserver` not `HTTPServer`
- no constant caps: `max_size` not `MAX_SIZE`

**only lowercase:**
- everything in `.sl` files must be lowercase
- this prevents agent hallucinations (camelcase vs snake_case)
- one correct way = zero ambiguity

---

### formatting

**indentation:**
- 2 spaces per level
- no tabs

**brackets:**
- opening bracket on same line: `fn main() {`
- closing bracket on new line aligned with statement start

**spacing:**
- spaces around operators: `x + y` not `x+y`
- space after comma: `fn(a, b, c)` not `fn(a,b,c)`
- no space inside parens: `(x + y)` not `( x + y )`

**line length:**
- no hard limit, use judgment
- break long lines at logical points
- align continued lines with opening delimiter

---

### strings

**quotes:**
- double quotes always: `"hello"` not `'hello'`
- no single quotes for strings

**interpolation:**
- use `$var` syntax: `"user: $username"`
- use `${expr}` for expressions: `"total: ${x + y}"`

**multiline:**
- triple quotes for multiline strings
- preserves formatting and newlines

---

### comments

**single line:**
- `#` for single line comments
- space after hash: `# this is a comment`

**doc comments:**
- `##` for documentation comments
- appears before function/bunch/mood definition
- becomes part of generated docs

**examples:**
```slop
## validates user credentials and returns auth token
fn authenticate(usr str, pwd str) (str, err)
  # check if user exists
  if !db.has(usr)
    ret ("", err{code: ecode.notfound, msg: "user not found"})
  
  # verify password
  if !verify(pwd)
    ret ("", err{code: ecode.unauthorized, msg: "invalid password"})
  
  (generatetoken(usr), nah)
```

---

### numbers

**decimal for:**
- ports: `6969`, `1337`, `4200`
- counts: `100`, `1000`
- ids: `42`, `69`, `1337`
- quantities: `67`, `420`

**hex for:**
- error codes: `0x01`, `0x10`, `0x20`
- addresses: `0xdeadbeef`, `0xcafebabe`
- flags/bitmasks: `0xff`, `0x00`
- aesthetic choices: `0x45`, `0x69`

## terminology

### og types

**definition:** built-in types that come with slop

**og types:**
- integers: `u8`, `u16`, `u32`, `u64`, `u128`, `u256`, `i8`, `i16`, `i32`, `i64`, `i128`, `i256`
- floats: `f32`, `f64`
- buffers: `b8`, `b16`, `b32`, `b64`, `b128`, `b256`, `bs`
- text: `str`, `ch`
- logic: `bool`
- special: `nah`

**usage in prose:**
- "og types" not "primitive types"
- "og" = original gangster (crypto culture reference)
- emphasizes these are the foundation types

---

### auto-spawned

**definition:** ffi binding wrapper code automatically generated by slopbind

**usage:**
- "auto-spawned bindings" for c/python/js wrapper code
- generated by `slopbind` tool
- wraps foreign functions for slop consumption

**examples:**
- c function binding wrappers
- python module bindings
- javascript interop code

**not for:**
- default methods (use "vibe-coded")
- compiler insertions (use "compiler-inserted")

---

### vibe-coded

**definition:** default methods that all types automatically get

**vibe-coded methods:**
- `.str()` - string representation
- `.json()` - json serialization
- `.hash()` - hash value
- `.isinit()` - initialization check
- `.clone()` - deep copy

**usage:**
- "vibe-coded methods" for built-in methods all types get
- these exist without explicit implementation
- compiler generates based on type structure

**not for:**
- ffi bindings (use "auto-spawned")
- manual implementations (just say "implemented")

---

### vibe-implemented

**definition:** when a bunch satisfies all methods required by a vibe

**usage:**
- "usr bunch is showable" means usr implements all showable methods
- "vibe-implemented" describes the relationship
- can be explicit (via `gives`) or implicit (bunch has all methods)

---

### slang terms

**deez:**
- instance reference in methods
- `deez.field` refers to current instance
- like `this` or `self` in other languages

**yeet:**
- fatal error termination
- `yeet ecode.unauthorized, "access denied"`
- logs error and exits process

**rekt:**
- cleanup/destruction of values
- automatically inserted by compiler before returns
- manual use for explicit cleanup

**nah:**
- void/nothing/empty value
- return type for void functions
- "no error" in error position

**bet:**
- confirmation/acknowledgment (in examples/comments)

**leak:**
- print to stdout
- `leak "value: $x"`
- primary debugging/output function

---

### mod terminology

**mod:**
- conventional name for modules
- not a keyword, just terminology
- "use mod" means import a module

**chunk:**
- compilation unit (a `.sl` file)
- used in error messages and jsonl output
- replaces generic term "file" in technical contexts

## file organization

### file structure order

**required order (0x01-0x0A):**

**0x01. arch declaration**
```slop
arch amd64 0.0.1
```

**0x02. imports**
```slop
use net
  http
  tcp
use crypto
  hash
```

**0x03. moods**
```slop
mood status u64
  pending = 0x00
  running = 0x01
```

**0x04. bunches**
```slop
bunch usr
  id u64
  name str
```

**0x05. gives (vibe implementations)**
```slop
gives usr showable
  fn str() str
    "usr:$deez.id:$deez.name"
```

**0x06. constants**
```slop
const max_users = 1000
const timeout = 30
```

**0x07. functions**
```slop
fn main()
  leak "yo world"
```

**0x08. tests**
```slop
test "user creation"
  u = usr{id: 1, name: "chad"}
  assert u.id == 1
```

**0x09. benchmarks**
```slop
bench "hash performance"
  data = "test data"
  hash(data)
```

**0x0A. inline assembly (if needed)**
```slop
asm amd64
  mov rax, 0x45
  ret
```

---

### main file rules

**executables:**
- must have `main.sl` file
- must have `main()` function in `main.sl`
- one `main()` per package maximum
- entry point for program execution

**libraries:**
- must not have `main.sl` file
- must not have `main()` function anywhere
- designed for import by other packages

**violations:**
- multiple `main()` functions → compiler error
- library with `main.sl` → not a library
- executable without `main()` → compiler error

---

### import formatting

**single mod:**
```slop
use net
```

**mod with specific items:**
```slop
use net
  http
  tcp
```

**multiple mods:**
```slop
use net
  http
use crypto
  hash
  sign
```

**ordering:**
- std mods first
- third-party mods second
- local mods last
- alphabetical within each group

---

### file naming conventions

**main files:**
- `main.sl` - entry point for executables

**common shortened names:**
- `srvr.sl` - server code
- `conf.sl` - configuration
- `util.sl` - utilities
- `db.sl` - database code
- `auth.sl` - authentication

**when to use full names:**
- when clarity is critical
- when shortened form is ambiguous
- when file is part of public api

## error conventions

### output formats

slop compiler supports two output formats for diagnostics:

**jsonl (default):**

```jsonl
{"lvl":"err","ecode":"0x20","chunk":"app.sl","ln":10,"col":1,"msg":"..."}
```

- stable format (recommended for tooling)
- one diagnostic per line
- standard json structure
- universal tool support (jq, grep, etc.)

**toon (experimental):**

```toon
app.sl[2]{ln,col,lvl,ecode,msg}:
  10,1,err,0x20,syntax error
  15,3,err,0x21,type error
```

- 40% fewer tokens vs jsonl
- tabular format for readability
- requires --experimental flag
- optimized for llm/agent consumption
- see [toon-format.md](toon-format.md) for details

### jsonl format

**standard format:**
```jsonl
{"lvl":"err","ecode":"0x01","chunk":"app.sl","ln":10,"col":1,"msg":"...","diff":"..."}
```

**required fields:**
- `lvl` - severity level ("err", "warn", "info")
- `ecode` - error code in hex ("0x01", "0x10", "0x20")
- `chunk` - source file name ("app.sl", "main.sl")
- `ln` - line number (1-indexed)
- `col` - column number (1-indexed)

**optional fields:**
- `msg` - human-readable message
- `diff` - suggested fix as unified diff
- `hint` - additional context or suggestion

**terminology:**
- use "ecode" not "code" (error code field)
- use "chunk" not "file" (source file field)
- this distinguishes from other code/file uses

---

### toon format (experimental)

**structure:**

diagnostics grouped by file in tabular format:

```toon
app.sl[2]{ln,col,lvl,ecode,msg}:
  10,1,err,0x20,syntax error
  15,3,err,0x21,type error

main.sl[1]{ln,col,lvl,ecode,msg}:
  30,5,warn,0x08,unused variable
```

**field order:** `ln,col,lvl,ecode,msg,diff` (location first)

**grouping:** diagnostics grouped by file to omit redundant chunk field

**sorting:**
- default: appearance order (--sort=location)
- optional: severity order (--sort=severity)

**quoting:** strings quoted when containing comma, colon, quotes, whitespace, or keywords

**benefits:**
- 40% token reduction vs jsonl
- human-readable grouped format
- eliminates redundant chunk field
- ideal for ai agent workflows

**enable:**

```bash
slopc main.sl --experimental --format=toon
```

**see:** [toon-format.md](toon-format.md) for complete specification

---

### error code categories

**info codes (icode):**
- `0x00` - none
- `0x01` - started
- `0x02` - stopped
- `0x03` - ready
- `0x04` - complete
- `0x05` - progress
- `0x06` - connected
- `0x07` - disconnected

**error codes (ecode):**
- `0x00` - none
- `0x01-0x0f` - not found/missing (0x01 notfound, 0x02 noent, 0x03 nofile)
- `0x10-0x1f` - permission/auth (0x10 unauthorized, 0x11 forbidden)
- `0x20-0x2f` - input/validation (0x20 invalid, 0x21 badreq, 0x22 badtype)
- `0x30-0x3f` - state/conflict (0x30 conflict, 0x31 exists)
- `0x40-0x4f` - i/o (0x40 ioerr, 0x41 readerr, 0x42 writeerr)
- `0x50-0x5f` - network (0x50 timeout, 0x51 unreachable, 0x52 refused)
- `0x60-0x6f` - parse/decode (0x60 parseerr, 0x61 decodeerr)
- `0x70-0x7f` - resource (0x70 nomem, 0x71 nospace, 0x72 toobig)
- `0xf0-0xff` - system/unknown (0xf0 syserr, 0xff unknown)

**warning codes (wcode):**
- `0x00` - none
- `0x01` - deprecated
- `0x02` - fallback
- `0x03` - slowpath
- `0x04` - retry
- `0x05` - partial
- `0x06` - truncated
- `0x07` - redundant
- `0x08` - unused
- `0x09` - experimental

see [errors.md](errors.md) for complete definitions and usage

## commit conventions

### commit message format

**structure:**
```
type(scope): short description

optional body text wrapped at 67 characters explaining the why
behind the change, not the what

optional footer with breaking changes, issue references
```

**types:**
- `feat` - new feature
- `fix` - bug fix
- `docs` - documentation only
- `style` - formatting, no code change
- `refactor` - code restructure, no behavior change
- `perf` - performance improvement
- `test` - add or modify tests
- `chore` - maintenance, tooling

**scope:**
- module name: `docs(errors)`, `feat(compiler)`
- feature area: `fix(memory)`, `perf(vibe)`
- omit if change is global: `docs: update readme`

**description:**
- lowercase, no period
- imperative mood: "add feature" not "added feature"
- 50 characters or less

**body:**
- wrap at 67 characters (not 72)
- explain why, not what
- reference design decisions
- separate from description with blank line

**footer:**
- breaking changes: `BREAKING: description`
- issue refs: `closes #123`, `fixes #456`
- separate from body with blank line

## port number conventions

### meme ports for examples

**valid ports (1024-65535):**
- `9110` - emergency vibes ("yo what's your emergency")
- `6969` - nice (primary http server examples)
- `6767` - repeating digits (variety)
- `1337` - leet (crypto/security examples)
- `4200` - blazeit (casual examples)
- `2077` - cyberpunk (futuristic vibes)

**usage by context:**

**http servers (primary):**
```slop
server = net.listen("0.0.0.0", 6969)
```

**emergency/error handling:**
```slop
fn emergency()
  leak "yo what's your emergency"
  listen(9110)
```

**crypto/security:**
```slop
node = blockchain.listen(1337)
```

**casual/variety:**
```slop
dev_server = http.listen(4200)
```

**cyberpunk/futuristic:**
```slop
neural_net = ai.listen(2077)
```

**repeating (variety):**
```slop
test_server = net.listen(6767)
```

### terminology consistency

**bunch vs struct:**
- always "bunch" never "struct"
- "bunch definition" not "struct definition"

**mood vs enum:**
- always "mood" never "enum"
- "mood values" not "enum variants"

**vibe vs trait/interface:**
- always "vibe" never "trait" or "interface"
- "implement vibe" not "satisfy trait"
